Built 2026-06-30 (dev server running; NO build run — read-only verification per the live-`.next` gotcha in [[live-market-data]]).

**EXPANDED CATEGORY TAXONOMY.** `src/modules/categories/defaultCategories.js` is the single source (used by seed + signup provisioning + backfill). Now has keys: food, groceries, shopping, fashion, bills, electricity, recharge, transport, fuel, entertainment, health, rent, emi, insurance, investment, services, transfer, income, other (each has label/emoji/color/kind/order). `categorize.js` `CATEGORY_STYLE` mirrors these (icon/iconBg) and `RULES` is ORDERED most-specific-first so new buckets win over broad ones: groceries before food; fashion before shopping; fuel before transport; electricity + recharge before bills; rent + emi + insurance + investment before services. Key merchant maps: bigbasket/blinkit/zepto/dmart→groceries; myntra/ajio/nykaa→fashion; hpcl/iocl/petrol→fuel; bescom/tatapower/electricity→electricity; airtel/jio/recharge/dth→recharge; lic/hdfc life/star health/policybazaar/term/health insurance→insurance; zerodha/groww/sip/mutual fund/nps/elss→investment.

**EXISTING-USER BACKFILL (how they get new categories).** `provisionUser.js` gained `backfillCategories(userId)` — idempotent insert of only the DEFAULT_CATEGORIES keys the user is MISSING (never overwrites customizations; ignores 11000 dup-key). Wired into `GET /api/categories` (self-healing: every category fetch tops up missing keys). `provisionNewUser` also calls it for existing users. seed.js now imports DEFAULT_CATEGORIES instead of a hardcoded list. Existing transactions with old keys still render (keys are additive; pickers + Reports look up by key with emoji/color fallbacks).

**SELF-TRANSFERS (Item A).** One-row model: `transaction.direction` enum gained `'transfer'`; new fields `toAccountId` + `toAccountName` (accountId = FROM/debited, toAccountId = TO/credited). `balance.service.js`: new `adjustAccount(userId, accountId, delta, trackSpend=false)`; the transfer branch in apply/reverse moves BOTH legs with trackSpend=FALSE (a transfer is NOT spending → never bumps cash spentThisMonth). Created as DRAFT (needs_review) → balances move only on confirm, same draft→confirm rule as everything else; delete reverses both legs (via the generalized deleteTransaction in [[goals-feature]]). EXCLUDED FROM REPORTS/SUMMARY automatically: every spend/income aggregation matches `direction:'debit'`/`'credit'` explicitly (getSummary, getCategoryBreakdown, report.service windowTotals/byCategory/byAccount/merchants/trend), so direction 'transfer' is ignored everywhere. POST /api/transactions schema allows direction 'transfer' + toAccountId with a refine() requiring distinct from/to. UI: TxnSheet has a 3rd "Transfer" tab (from + to account pickers, merchant auto = "From → To", categoryKey 'transfer', icon 🔁); Transactions list shows transfers neutrally (no +/- sign, "Transfer" subtitle) + a 🔁 Transfer quick-add button. listTransactions segment in/out naturally exclude transfers.

**RECURRING / MONTHLY EXPENSES (Item D).** `transaction` gained `recurring` (Bool, indexed) + `frequency` ('monthly'). NO auto-generation of future rows (lightweight per spec) — just the flag. TxnSheet expense mode has a "Monthly recurring" toggle. Transactions list shows a 🔁 MONTHLY pill on recurring rows + a "🔁 Monthly bills" filter chip (client-side filter on txns.data via recurringOnly state). New categories insurance 🛡️ + investment 📈 cover SIP/term/health/life insurance.

**addedToday for reminder.** getSummary now also returns `addedToday` (count of txns created OR occurring since local midnight, any status) — powers the reminder banner (see [[reminder-and-manual-investments]]).
