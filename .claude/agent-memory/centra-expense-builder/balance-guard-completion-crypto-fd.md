**ITEM 1 — NO NEGATIVE BALANCE / OVERDRAW GUARD (user-initiated only).** Server: `transaction.service.assertSufficientBalance(userId,{accountId,amount})` throws 400 `Amount exceeds <account> balance (₹X available).` `isUserInitiatedOutflow(data)` = source manual|cash AND direction debit|transfer (covers manual expense, transfer FROM-leg, goal contributions which are source manual/cash debit). Email/aa_sync NEVER blocked. Checked at BOTH `createTransaction` (immediate feedback; goal contributions apply to goal.saved on create so must be guarded then) and `confirmTransaction` (re-validate a draft at confirm — balance may have dropped since). New `balance.service.getAccountBalance` returns {balance,name}. DISPLAY CLAMP: new `format.inrBalance(n)` = inr(max(0,n)) — used for every account-balance display (Dashboard cards+totals, Profile, CashSheet, OnboardingWizard); Dashboard totals computed from clamp0'd per-account balances so a negative computed balance can't drag the visible total down. Stored balance stays honest (email-driven could theoretically go negative); only the display clamps. UI hints: TxnSheet (expense+transfer) and ContributeSheet show "₹X available in <acct>", turn red + disable submit ("Insufficient balance") when amount>available; server re-validates.

**ITEM 2 — GOAL COMPLETION + ACTIVE/COMPLETED SPLIT + CELEBRATION.** goal.model gained `completedAt` (Date|null). goal.service `syncCompletion(goal)` (no save) sets completedAt first time saved>=target, clears it if it drops back below; called in updateGoal + applyGoalForTransaction + reverseGoalForTransaction. GoalsSection: `isDone(g)`=completedAt||saved>=target; splits ACTIVE vs COMPLETED sections; completed cards get a green ✓ ACHIEVED badge + green full bar + "🎉 Goal achieved! You saved ₹X for <name>." One-time celebration TOAST: a `completedIds` useRef tracks the prior completed set across refetches; a goal newly crossing → green toast (auto-dismiss 4.5s, uses existing fadeIn keyframe). Over-funding still allowed (not clamped); completion just stops at >=100%.

**ITEM 3 — MANUAL CRYPTO (USD, qty, cost, date, live P/L).** holding.model gained `currency` ('INR'|'USD'), `coinId`, `buyPriceUsd`, `purchaseDate` (+reused `units`). crypto.client: `getCryptoPrices` now also returns `id`; new `getSpotPricesForIds(ids)` → {id:usd} (per-id 45s cache, serves-stale/null on fail, never throws). holding.service `valueHoldings(holdings)` recomputes crypto currentValue = units×spotUSD live (sets spotUsd + priceStale; falls back to stored on fail). createHolding/updateHolding set currency USD + cost basis = units×buyPriceUsd. AddHoldingSheet crypto branch: coin picker (from api.getCrypto, shows live $), quantity, buy price USD, purchase date + cost-basis/live preview. Investments.jsx manual rows are currency-aware via `money(h,v)` (=fmtPrice USD for crypto else inr); crypto shows units/cost/spot/since + $ LIVE / PRICE STALE pill + P/L% .

**ITEM 4 — FD with quarterly maturity + auto-credit.** holding.model gained `principal`, `fdStartDate`, `maturityDate`(existing), `creditAccountId`, `maturityValue`, `maturedCredited`. Formula (USER DECISION quarterly): `fdMaturityValue = round(principal × (1+rate/400)^(4×years))`, years from start→maturity. `fdAccruedValue` = value to-date (capped at maturity). AUTO-CREDIT: `processMaturedFDs(userId)` — for manual FDs past maturityDate with maturedCredited!=true: flips maturedCredited FIRST (idempotency), then posts a CONFIRMED income credit (merchant "FD Maturity: <name>", category income) to creditAccountId via createTransaction → updates that account balance. TRIGGER (no cron): called lazily at the start of BOTH `GET /api/holdings` and `holdings.getPortfolio` (so it runs on app load when Investments fetches). AddHoldingSheet FD branch: name/principal/rate/start/tenure-months(→maturity)/credit-account + quarterly maturity preview. Investments rows show principal/rate/start→maturity/≈maturity + "✓ MATURED · CREDITED" pill.
