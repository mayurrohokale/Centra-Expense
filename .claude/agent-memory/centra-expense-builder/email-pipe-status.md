**Key decisions / behaviors:**
- **Single-inbox model** (kind='gmail', accountId=null) scans ALL bank senders — a DEVIATION from M1's per-bank-account "Connect email" cards (design showed per-account linking). Build spec explicitly redefined this screen, so functional spec won over the M1 stub. Auto-detected txns still grouped by bank in the UI.
- Account match: parsed last4 → bank name token → else create lightweight bank account. Demo seed last4s (HDFC 4582 / ICICI 9013 / SBI 7745 / Axis 3320 / Kotak 1198) match the fixtures so they link by last4.
- Fingerprint = `fingerprint([userId, bankKey, last4, amount, occurredAtISO, emailId])` — emailId/gmail messageId is used ONLY here, never stored. Dedupe verified: simulate #1 = 10 created, #2 = 0 created / 10 duplicate. Seed's 2 pre-existing email needs_review rows use a different fp formula so they don't collide (expected, not a bug).
- All email txns: source='email', status='needs_review'. Confirmed: transaction docs contain ZERO body/raw/html/messageId/long-string fields; connection default projection excludes encryptedTokens.
- gmail/start with no Google creds → 307 redirect to `/?tab=email&gmail=unconfigured` (no 500). Cron rejects without/with-wrong secret (401), 200 with correct Bearer. Simulate guard: blocked only when `isProd && gmailConfigured`.

**MULTI-GMAIL (2026-06-30):** A user can now connect N Gmail inboxes. Extended the existing `connections` collection (NOT a new model — reuses encryption/revoke/cron). New fields: `emailAddress` (per-account identity), `tokenExpiresAt`, `historyId`. Partial UNIQUE index on `(userId, kind, emailAddress)` where `emailAddress > ''` (AA exempt). `upsertConnection` now keys email kinds on emailAddress (was `(userId,kind,accountId=null)` → that OVERWROTE the 2nd inbox). New: `syncAllGmailForUser`, `POST /api/connections/sync-all`, `listConnectionsByKind`, `markConnectionError`. Disconnect now also best-effort calls Google `/revoke`. EmailConnect.jsx renders one card per inbox + "Connect another" + "Sync all".
- **Dedup formula CHANGED**: fingerprint dropped the Gmail messageId; now content-only `[userId, bankKey, last4, amount, occurredAtISO, direction, merchant]` so the SAME alert in two inboxes dedupes. Trade-off: two identical-looking distinct txns merge (rare, accepted). NOTE: live emails ingested under the OLD (messageId) formula will re-create ONCE on next sync after this change.
- **Sync returns 0 on personal inboxes (BY DESIGN):** the Gmail query is `from:(hdfcbank.net OR hdfcbank.com OR icicibank.com OR sbi.co.in OR alerts.sbi.co.in OR sbicard.com OR axisbank.com OR kotak.com OR kotakbank.com OR kotak.net) newer_than:Nd` (built in `gmailClient.buildBankQuery`). An inbox with no mail from those 5 banks legitimately matches 0 → `total:0`. First-run window widened 30→90d (`FIRST_RUN_BACKFILL_DAYS`); incremental after = days-since-lastSync. `syncGmailConnection` now logs `[gmail-sync]` (q, candidate count, read-access probe via `countRecentMessages` when 0, fetched count, created/dupe/failed). `total`=candidates fetched; `failed`=fetched-but-no-regex-match. Test without real alerts: Simulate fetch (10 fixtures) OR self-send a matching alert. ANTHROPIC_API_KEY won't help find emails (parse fallback only, after a match).
- **Bank domains modernized + 3-stage extraction (2026-06-30):** `bankSenders.js` now has 12 banks (added PNB/BoB/Yes/IndusInd/IDFC First/Canara/Union) and each lists BOTH legacy domains AND the RBI `.bank.in` domain (e.g. HDFC real sender is `alerts@hdfcbank.bank.in` — old config of only hdfcbank.net/.com missed it → the real cause of "HDFC connected but 0 synced"). `bankForSender` is now boundary-safe (`domain===d || endsWith('.'+d)`) so any local-part/subdomain matches but `myaxisbank.com` doesn't. Extraction chain in `parsers/registry.js` `parseEmail` (now ASYNC): per-bank regex → `parsers/generic.parser.js` `genericExtract` (bank-agnostic amount/direction/last4/date/merchant) → `aiExtract.js` (Anthropic). Banks without a per-bank parser (PNB etc.) still work via generic+AI. `ingestEmails` awaits parseEmail and logs `[gmail-sync] <bank>: extracted via regex|generic|ai → <dir> ₹<amt>`.
- **AI fallback wiring:** `aiExtract.js` calls Messages API via the existing `httpClient` (NOT the `@anthropic-ai/sdk` — avoided adding an unapproved dep; flagged SDK as optional swap). Strict JSON via `output_config.format` json_schema (prefill is removed on Opus 4.8); zod-validates the response; graceful no-op returning null when `ANTHROPIC_API_KEY` unset. NO `thinking`/`effort` sent so it works on Haiku too (effort 400s on Haiku-4.5). Model = `env.anthropicModel` (new), default `claude-opus-4-8`; set `ANTHROPIC_MODEL=claude-haiku-4-5` for cheap extraction. AI only fires when regex+generic both fail.
- **Merchant auto-categorization (2026-06-30):** `categorize.js` strengthened into an ordered rule table (`RULES`) + `normalizeMerchant` (strips VPA/@handle/punct) mapping to the EXISTING 9 category keys only (groceries→food, fuel→transport, education/investments/rent→services, ATM/transfers→other). `categorize()` now returns `categorySource` ('merchant-rule'|'default'); new tx model field `categorySource` enum ['merchant-rule','ai','default','manual']. Optional AI: `aiSuggestCategory` in aiExtract.js (strict-JSON enum, no-op without key) fires only for default 'other' debits. Review UI (EmailConnect.jsx): per-card category chip (tap→CategoryPickerSheet), AUTO/AI badge for high-confidence, "Confirm N auto-categorized" bulk button (keeps each auto category). Taxonomy = seed.js categories + CATEGORY_STYLE/CATEGORY_KEYS in categorize.js.
- **Live OAuth debugging seam:** callback `app/api/connections/gmail/callback/route.js` now logs `[gmail-callback]` at each step, distinguishes state-fail reasons, guards the save (redirects `gmail=save_failed` not 500), and surfaces Google's error body (`invalid_client`/`invalid_grant`/`redirect_uri_mismatch`) via `googleErrorDetail` in gmailOAuth.js. redirect_uri is identical in auth + token-exchange (both `env.gmailRedirectUri`).
